## Verified now

- Rust monorepo scaffold exists.
- `domain` crate implements Profile IR v0, artifacts, secret references, signed runner commands, signed deployment results, deployment result/plan, deployed profile shape.
- `compiler-xray` compiles VLESS + REALITY, Shadowsocks, and Trojan Profile IR to Xray JSON and rejects REALITY on `xray-core < 1.8.0`.
- `compiler-xray` validates Shadowsocks 2022 PSK shape before runner time; Docker-backed real Xray smoke verifies compiled Shadowsocks output and LocalRunner apply with `xray run -test`.
- `runner` verifies signed commands and applies configs through temp write, `xray run -test`, atomic active symlink switch, optional reload/restart hook, and optional process health hook. If validation/reload/health fails and a previous active release exists, it reports `RolledBack` and preserves the previous active release. It also executes structured `RollbackDeployment` commands by switching `active` back to a previous local release. The runner binary supports HTTP outbound polling via `CONTROL_PLANE_BASE_URL`, can self-register with `NODE_REGISTRATION_TOKEN`, signs deployment results with its node key, and submits them to the control-plane. `RUNNER_RESULT_SIGNING_KEY_HEX` can override the dev key; per-node runner result public keys can be rotated through the control-plane.
- `subscription` generates grouped subscription artifacts and hides revoked credentials; the control-plane subscription view also suppresses quota-exceeded and expired credentials.
- `control-plane` exposes minimal local API for one-time-token node registration, runner result key rotation, authenticated runner heartbeat, VLESS/SS/Trojan profile and client creation, client quota/expiry decision lookup, subscription token issue/rotation, idempotent compile/deploy requests, runner command polling/result submission, raw usage sample reporting/query, hourly/daily/monthly usage rollup query, deployment status, deployment health lookup, multi-sample deployment readiness, rollout advance promotion/auto-rollback, rollback pointer lookup, deployment snapshot lookup, operator rollback command enqueue, and subscription retrieval. Node registration can persist `runner_result_public_key_hex`, and result submission is verified against that per-node key. Runner heartbeat/poll/result/usage/health endpoints require `X-Runner-Token`.
- `storage` crate now owns repository records, an async `ProxyStore` trait, `MemoryStore`, and `PostgresStore` sqlx read/write paths for the P0 flow, including runner command queue, deployment result status updates, runner result key rotation, deployment health sample checks, rollback pointers, deployment snapshots, raw usage samples, hourly/daily/monthly usage rollups, credential quota/expiry decisions, idempotency responses, subscription token rotation/access logs, and content-addressed artifact blobs with sha256 mismatch rejection.
- P0-shaped Postgres migration exists in `migrations/0001_p0_schema.sql`, including `idempotency_keys`, `profile_credentials` for profile-to-credential projection, `credential_quotas` and `credentials.expires_at` for quota/expiry decisions, `subscription_tokens`/`subscription_access_logs`, and `rollback_pointers` for deployment rollback evidence. Docker-backed Postgres smoke verification now runs the migration against a disposable Postgres 16 database and exercises the PostgresStore P0 path.
- Operational docs, ADRs, `.env.example`, `justfile`, and a verified Next.js App Router console exist. Console pages for `/dashboard`, `/nodes`, `/clients`, `/profiles`, `/deployments`, `/tasks`, `/logs`, and `/settings` now contain concrete P0 runbook cards and command blocks instead of backend-only placeholders.

## Known remaining production hardening / post-P0 gaps

- Control-plane now uses the `storage::ProxyStore` trait and can select `PostgresStore` when `DATABASE_URL` is set. Live Postgres migration/integration execution is verified through `./scripts/postgres_smoke.sh` using Docker and Postgres 16.
- Runner has a tested outbound polling abstraction and binary HTTP polling mode with `RUNNER_API_TOKEN` -> `X-Runner-Token`. On startup it can self-register with `NODE_REGISTRATION_TOKEN` and then posts a capability heartbeat containing runner version, work dir, and xray binary path. Deployment results are Ed25519-signed in the HTTP path and verified against registered per-node public keys. Production hardening such as mTLS, richer key rotation UX, backoff, and durable session tracking remains pending.
- Control-plane now dispatches signed deployment commands through `ProxyStore`, accepts runner results, and exposes deployment status. The migration includes `runner_commands`, and Postgres-backed command queue execution is covered by the Docker smoke test.
- Local `xray` binary is not installed, but real Xray config validation is covered by `./scripts/xray_smoke.sh` and `./scripts/runner_xray_smoke.sh` using Docker and `ghcr.io/xtls/xray-core:latest`; runner filesystem/symlink behavior is also covered with fake local xray tests.
- `pnpm` is not installed on this machine, so the web console was installed and verified with `npm --prefix apps/web ...`; `apps/web/package-lock.json` records the resolved local toolchain.
- Multi-sample health-gated readiness and P0 rollout advance are implemented. `GET /deployments/{deployment_id}/readiness` requires a successful deployment result plus two healthy samples, and blocks on a latest unhealthy sample. `POST /deployments/{deployment_id}/advance` promotes ready deployments and automatically queues rollback when the latest health sample is unhealthy. More advanced multi-node/canary traffic shifting is still pending. Basic runner result key rotation UX, quota/expiry decisions from accumulated usage rollups and `credentials.expires_at`, quota/expiry-aware subscription suppression, subscription token rotation/access logging, deployment health checks, raw usage samples, hourly/daily/monthly usage rollups, per-node runner result keys, and operator-triggered rollback command orchestration are implemented.


Additional rollback evidence: `cargo test -p runner --test runner_contract bad_config_fails_and_keeps_previous_active_release` verifies an invalid config after a successful release returns `RolledBack` and leaves `active` pointing to the previous release.
